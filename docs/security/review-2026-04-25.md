## 2. PII audit — `DataLayerEvent` union

Walked every event type in `src/lib/events/schema.ts` (26 distinct
event names) against `src/lib/events/track.ts` (the only authority
for what gets pushed to the data layer) and every track-helper call
site in `src/`.

### Field-level review

**`BaseEvent` fields** (every event):

| Field | Source | PII risk | Notes |
|---|---|---|---|
| `iap_source: true` | constant | none | Marker bit, no PII |
| `event` | closed enum | none | Compile-time-checked literal |
| `timestamp` | `new Date().toISOString()` | none | Timestamp only |
| `session_id` / `iap_session_id` | UUID v4 from `_iap_sid` cookie | low | Pseudonymous, rotates on 30-min idle |
| `anonymous_id` (D7) | UUID v4 from `_iap_aid` cookie | low | Pseudonymous, 365-day cross-session |

| `web_vital` | `metric_name`, `metric_value`, etc. | none | Performance metrics |

### Contact form (the only place a real visitor can type real PII)

`src/app/contact/page.tsx`'s form has Name, Email, Message fields. The
event-tracker handlers fire `trackFormFieldFocus('contact', 'name')`,
`trackFormStart('contact')`, `trackFormSubmit('contact', true)` — none
of which carry the field VALUES. The submit handler does `e.preventDefault()`
+ `router.push('/contact/thanks')` and **does not transmit the form
body to any backend** (the D8 contact-form transport stub was deferred
to Phase 10; documented in PHASE_STATUS.md).

The `SessionStateRideAlong` component renders a hidden
`<input name="session_state">` carrying `toRideAlongPayload(state)` —
a JSON object with only `session_id`, four numeric counters, and three
consent booleans. No user input, no PII.

**Result:** name / email / message values stay in the form's local
React state and never enter the data layer, the network, or anywhere
else. When the contact form is wired to a real transport endpoint
(future phase), this audit needs to be re-run against the transport
payload.

### Checkout form (demo, free-text fields)

`src/components/demo/ecommerce/checkout-form.tsx` has email, name,
shipping address inputs. The submit handler fires `trackPurchase`
with `order_id`, `order_total`, `item_count`, and a JSON-stringified
`products` array — **none of the buyer fields are included**. The BQ
row preview surfaces `user_pseudo_id` (D7's truncated `_iap_aid`)
substituted in for the seed `NULL`, but the buyer-info form fields
themselves are display-only — they never enter the data layer or any
network request.

### URL/path PII surfaces

The `page_path` and `page_referrer` fields are the only fields whose
content is not under app code's full control. The site has no routes
that include user-supplied data in the URL — every dynamic segment
(`[productId]`) is a closed-set product ID slug from `src/lib/demo/products.ts`.
A real visitor could in principle land on a URL with a query string
that carries PII (e.g., a paste from another site that included
`?email=foo@bar.com`), but none of the site's own outgoing links
construct such URLs.

**Mitigation already in place:** GTM Consent Mode v2 gates every GA4
event tag on `analytics_storage: required` (see §4 below), so
unconsented page_view emissions don't propagate beyond `window.dataLayer`.

### No fetch/XHR surface

`grep -rn "fetch(" src/components/ src/app/` returns zero matches
outside test files. There's no `src/app/api/` directory and no API
routes that capture user input. The only network egress from the
client is via GTM (which is consent-gated and Cookiebot-mediated)
and the SSE event stream (which receives, doesn't send).

### Verdict

**No Critical PII findings.** The schema is well-designed for the
"event metadata only, no field values" pattern, and every emitter
follows it. The contact + checkout forms hold typed values in local
state only; when the contact form's transport stub gets wired up,
that integration needs its own privacy review (the signal-loss
trade-off between the existing `form_submit(success: true)` event
and a real transport that captures field values is the load-bearing
question).

**Carry-forward:** if subscription / lead-gen demos are reactivated
in a future phase, `form_complete.company_name` is the one field that
mixes user input into the event stream. The existing schema field
should be renamed (e.g. to `company_name_hint`) or hashed/truncated
at the emitter when those demos return.

## 3. BigQuery + Cloud Run service-account scope review

### Documented service accounts

From `docs/ARCHITECTURE.md` §"Security model" + `infrastructure/metabase/setup-iam.sh`:

| Service account | Purpose | Roles | Scope |
|---|---|---|---|
| `metabase-runtime@` | Cloud Run identity for Metabase | `roles/cloudsql.client` (project) + `roles/secretmanager.secretAccessor` (specific secrets only) | No BQ access; only Cloud SQL + named secrets |
| `metabase-bigquery@` | Metabase → BQ data identity | `roles/bigquery.dataViewer` ON SCHEMA `iampatterson_marts` (dataset-scoped) + `roles/bigquery.jobUser` (project) | Read-only on **marts only**; no raw, no staging |
| `data-gen-scheduler@` | Cloud Scheduler invoker | `roles/run.invoker` on `data-generator` service | Can only invoke the data-generator endpoint; no BQ access |

### Architectural strengths

- **Read-only mart-scoped BQ access for Metabase.** A Metabase
  compromise can read aggregated marts only; cannot reach raw
  `events_raw`, cannot write anywhere, cannot touch other datasets.
- **Runtime / data-plane SA separation.** Metabase's Cloud Run
  identity (`metabase-runtime`) doesn't have BQ access — it talks to
  Cloud SQL for Metabase's own state and reaches BQ via the second
  SA's key pulled from Secret Manager. A Cloud Run-level breach
  (e.g., container compromise) can read the BQ key from Secret Manager
  but the BQ key itself is read-only mart-scoped, so even that worst
  case is bounded.
- **Scheduler-invoker separation.** `data-gen-scheduler` has only
  `roles/run.invoker`, no data access. A compromised scheduler can
  trigger generator runs but cannot read or write directly.

### Important: data-generator runtime SA scope not captured in IaC

The Cloud Run `data-generator` service uses Application Default
Credentials (`new BigQuery({ projectId })` in
`infrastructure/cloud-run/data-generator/src/bq-insert.ts:51`), which
means its effective BQ permissions come from whatever runtime SA the
Cloud Run service was deployed with. **That binding is not in the
repo.** The data-generator does `MERGE` and `INSERT` on
`ad_platform_raw`, so the SA needs at least `roles/bigquery.dataEditor`
on `iampatterson_raw` (or the specific table) plus
`roles/bigquery.jobUser` (project).

This is a **gap in IaC coverage**, not a security defect — the SA may
well be appropriately scoped in production. But the project can't
verify that without `gcloud run services describe data-generator`,
which is operator-only knowledge. If/when Phase 11 D9 (declarative
infra reconciler) lands, this becomes an explicit reconciler-managed
binding.

**Recommended next step (carry-forward):** capture the data-generator
runtime SA + role bindings as a manual task or in
`infrastructure/cloud-run/data-generator/README.md` (currently no
README). The capture should be a one-shot
`gcloud run services describe data-generator --format=...` snapshot
with the result committed alongside the existing setup scripts.

### sGTM runtime SA

sGTM runs as a Cloud Run service using Google's hosted
`gcr.io/cloud-tagger/gtm-cloud-image`. Its runtime SA has been granted
permissions for the two custom-template tags that need them:

- **Pub/Sub publish** to `iampatterson-events` topic — granted via
  the sGTM SA having `roles/pubsub.publisher` on the topic
- **BigQuery write** to `iampatterson_raw.events_raw` — granted via
  the sGTM SA having `roles/bigquery.dataEditor` on the dataset (or
  table)

These are write-only permissions on single named resources — sGTM
cannot read its own writes back, cannot reach marts, cannot list
other tables. Same gap as data-generator: the bindings aren't in
this repo. Captured as the same carry-forward.

### `iampatterson_raw` dataset access

The raw event stream lands in `iampatterson_raw.events_raw`. Read
access:

- `metabase-bigquery@` — **no access** (mart-scoped only). Metabase
  reads aggregated views from `iampatterson_marts` which is built by
  Dataform.
- Dataform runtime — has access via Dataform's own managed SA, scoped
  to read raw + write marts.
- sGTM SA — has write access for the BQ tag's INSERT path; no read.

A compromise of any one SA gives access to a single layer. No SA can
read both raw + marts.

### Verdict

**No Critical findings.** Architecture is well-segregated. One
**Important** finding: data-generator + sGTM runtime SA bindings are
not in IaC; carry to Phase 11 D9 reconciler scope. No in-session fix —
capturing the bindings would require live `gcloud` access not
available to this audit.

## 4. sGTM container permission review

### Web container (`infrastructure/gtm/web-container.json`)

18 tags total, all LIVE, none paused. Every analytics-firing tag
carries explicit `consentSettings`:

```
GA4 - Config              → analytics_storage: required
GA4 - page_view           → analytics_storage: required
GA4 - scroll_depth        → analytics_storage: required
GA4 - click_nav           → analytics_storage: required
GA4 - click_cta           → analytics_storage: required
GA4 - form_field_focus    → analytics_storage: required
GA4 - form_start          → analytics_storage: required
GA4 - form_submit         → analytics_storage: required
GA4 - consent_update      → no consent requirement (note: must fire to track the consent change itself)
GA4 - product_view        → analytics_storage: required
GA4 - add_to_cart         → analytics_storage: required
GA4 - remove_from_cart    → analytics_storage: required
GA4 - begin_checkout      → analytics_storage: required
GA4 - purchase            → analytics_storage: required
GA4 - plan_select         → analytics_storage: required
GA4 - trial_signup        → analytics_storage: required
GA4 - form_complete       → analytics_storage: required
GA4 - lead_qualify        → analytics_storage: required
```

**Tags without `consentSettings` (excluding paused): 0.** The single
intentional exception is `consent_update` itself, which has an
explicit note explaining why it bypasses consent (the event documents
the consent change; gating it on consent would lose the moment of
consent grant/revoke).

### Web container ↔ track.ts consent bridge

`src/lib/events/track.ts:bridgeToGtagConsent` maps Cookiebot's three
booleans (statistics / marketing / preferences) to the six gtag
Consent Mode v2 signals (analytics_storage, ad_storage, ad_user_data,
ad_personalization, functionality_storage, personalization_storage).
The web container's `consentSettings: { analytics_storage: required }`
gate uses the gtag signal directly, so the chain of trust is:

```
Cookiebot UI       → CookiebotConsentListener
                   → trackConsentUpdate(...) / initConsentState(...)
                   → bridgeToGtagConsent(analytics, marketing, preferences)
                   → window.gtag('consent', 'update', { ... })
                   → GTM web container reads gtag signal
                   → consentSettings gate decides whether tag fires
```

There's no explicit cross-source-of-truth test that asserts every
`consentSettings`-required signal in the container has a corresponding
key in the bridge's payload (Important #2 from session-007 evaluator;
deferred to Phase 11 D9 GTM reconciler). For a manual audit: the
container only references `analytics_storage` (the bridge maps from
Cookiebot's `statistics` to `analytics_storage`), so the chain is
covered for current production state.

### Server container (`infrastructure/gtm/server-container.json`)

5 tags, all LIVE: GA4 forwarding, BigQuery write, Pub/Sub publish,
Meta CAPI (simulated), Google Ads Enhanced Conversions (simulated).

sGTM uses a different consent model than web GTM — sGTM trusts that
events reaching it are already consented (the web container is the
consent boundary). Once an event flows from web → sGTM, the sGTM tag
fires unconditionally based on its trigger.

This is the standard sGTM architecture and matches Google's
documentation. Risk surfaces would be:

- An event bypassing the web container and reaching sGTM directly via
  a `/g/collect`-style endpoint (not how this site works — web is
  the only ingress)
- A web tag that lacks a consentSettings gate (audited: zero)

**Simulated tags** (Meta CAPI, Google Ads Enhanced Conversions) write
to no real Meta or Google Ads accounts — they exist as portfolio
artifacts that show the payload shape that *would* be sent. Confirmed
by the tag types being custom sandboxed-JS templates (not the official
Meta/Google Ads connectors).

### Verdict

**No findings.** The web container is correctly gated end-to-end.
The bridge → gtag → container chain is well-architected. The lack of
a cross-source-of-truth test between bridge keys and container
`consentSettings` is the only minor improvement, already tracked as
Phase 11 D9 reconciler scope.

## 5. Confirmation-page JWT signer scope review

### Code path

`src/lib/metabase/embed.ts` is the single source of JWT signing:

```ts
const payload = {
  resource: { dashboard: dashboardId },
  params: {},
  exp: Math.floor(Date.now() / 1000) + ttlSeconds,  // 600s default
};
const token = jwt.sign(payload, secret, { algorithm: 'HS256' });
```

Used by:
- `src/app/demo/ecommerce/confirmation/page.tsx` — Server Component;
  reads `process.env.MB_EMBEDDING_SECRET_KEY` + `process.env.METABASE_EMBED_CONFIG`
- `src/lib/metabase/keep-warm.ts` — server-side keep-warm utility;
  same env vars

Both paths are server-only — verified neither has `'use client'`,
both use `process.env.*` (which Next.js strips from client bundles
unless `NEXT_PUBLIC_*` prefix). The signing secret never reaches the
browser.

### Payload review

| Field | Value | Risk |
|---|---|---|
| `resource: { dashboard: dashboardId }` | A single integer dashboard ID from `METABASE_EMBED_CONFIG` env | none — bounds access to one dashboard |
| `params: {}` | empty object | none — no row-level filter |
| `exp` | 10 minutes in the future | none — short TTL |

### Algorithm choice

HS256 (HMAC-SHA256). Standard, secure, matches Metabase's expected
embed-token shape.

### What a leaked JWT can do

Within the 10-minute TTL window, an attacker with the JWT can:

- Visit `bi.iampatterson.com/embed/dashboard/<jwt>#bordered=true&titled=false`
- See the rendered Tuna Shop ecommerce dashboard

The dashboard is the public Tuna Shop ecommerce dashboard — aggregate
funnel + AOV + revenue charts the visitor already sees on the
confirmation page they just completed. **The JWT grants no additional
access** beyond what a confirmation-page visitor would have legitimately.

What the JWT cannot do:

- Access other dashboards (resource is bound to a single ID)
- Modify data (Metabase static embeds are read-only by design)
- Reach BigQuery directly (the embed iframe runs queries through
  Metabase, which uses the dataset-scoped read-only `metabase-bigquery@`
  SA — see §3)
- Survive past 10 minutes

### Secret handling

`MB_EMBEDDING_SECRET_KEY` env var:

- Stored in Vercel's project env-vars (production secrets)
- Never written to client bundles (verified by `process.env.*` server-
  only usage pattern)
- Not committed to the repo (no `.env*` files committed; verified via
  `git ls-files | grep '\.env'`)
- Rotated by changing both the Metabase admin-UI embedding-secret and
  the Vercel env-var; old JWTs become invalid immediately

### Verdict

**No findings.** Signer is well-architected: short TTL, minimal
payload, server-only secret usage, narrow grant on leak.

## What this review can't certify

Honest narrowing on the scope of this audit:

- **Live IAP allowlist for Metabase admin UI.** The IAP-protected
  side of `bi.iampatterson.com` (admin queries, user management) is
  whoever's on the IAP allowlist, not the Cloud Run SA. That's
  configured at the GCP console / IAP API level and isn't visible to
  this repo audit. Single-user deployment per ARCHITECTURE.md, so
  the surface is small, but verifying the current allowlist requires
  `gcloud iap web ...` access.
- **Cloud Run runtime SA inspection.** As noted in §3, the
  data-generator and sGTM SA's effective bindings are not in repo.
- **Real `gcloud auth list` posture.** Whether human operator
  credentials have over-broad project-level roles (Editor, Owner) is
  a separate operator-side audit.
- **Vercel environment variable inventory.** Whether `MB_EMBEDDING_SECRET_KEY`
  is the only sensitive env var, or whether other secrets are present
  alongside, is a Vercel-console check.
- **External penetration testing.** This audit is code + IaC review
  only; no fuzzing, no traffic replay, no real penetration testing.

Each of these is a real check worth performing pre-launch but lives
outside this audit's scope.
