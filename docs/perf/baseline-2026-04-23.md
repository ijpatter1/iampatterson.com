**Readout:**

- **Desktop is already shipping-ready perf.** Perf 99–100 across the board, LCP 0.7–0.8s, everything green. The hero h1 is correctly the LCP element on `/` (1072×474 at 139,158) and on `/demo/ecommerce/confirmation`; on `/demo/ecommerce` the LCP element is the walkthrough body section. No desktop CWV work is required for 10b — the numbers are a portfolio-grade.
- **Mobile is the real optimization surface.** Mobile LCP 5.4s (home) / 8.2s (confirmation) is where the D1c levers live. The slowdown is CPU + network throttling — Lighthouse emulates a mid-range phone on a 1.6Mbps Slow-4G link with 4× CPU multiplier.
- **Mobile LCP element is not the hero h1.** On a 360px viewport, `clamp(44px, 13vw, 200px)` resolves to 46.8px, giving the h1 a bounding area just smaller than the supporting paragraph (~55K vs ~56K sq px). The paragraph wins mobile LCP by a hair. **On desktop the h1 wins decisively** (~442K sq px).
- **CLS is already excellent** across both presets (~0). No D1c work needed there.
- **INP is not a Lighthouse audit** (field-only metric). The `WebVitalsReporter` landed in D1a captures real INP numbers once the sGTM trigger + BQ tag catch up. `max-potential-fid` (Lighthouse's synthetic proxy) sits in the "good" band across all runs.
- **Mobile `/demo/ecommerce` shows Perf=0** from a non-fatal `@paulirish/trace_engine` insight-computation error during the run. The `after()` hook's background fetch to `bi.iampatterson.com` likely trips the synthetic trace engine under throttled mobile emulation — same class of quirk the Phase 10a smoke notes for `waitUntil: 'networkidle'`. Desktop preset runs the same route cleanly at Perf=100, so this is purely a synthetic mobile-throttle interaction, not a real-user issue.
- **Confirmation's BP score (78 mobile / 78 desktop vs 100 elsewhere)** is the one cross-preset regression worth investigating during D1c.

**Mobile render-delay anatomy** — the home-page mobile LCP is 5.4s with Cookiebot and 3.8s without (rebuild with `NEXT_PUBLIC_COOKIEBOT_ID=""`). The gap, ~1.6s, is Cookiebot-attributable (uc.js + cc.js = 141KB on Slow 4G ≈ 700ms download + parse + execute on throttled CPU). The remaining 3.3s without Cookiebot is fonts (243KB across 4 files ≈ 1.2s download) + React hydration on throttled CPU + CSS parse + paint. Not one big lever — several smaller ones.

## D1c — Investigations

### Finding: home LCP element is the Cookiebot consent dialog, not the hero

Inspecting `lighthouse-2026-04-23-home.report.json` → `audits['largest-contentful-paint-element']`:

```
selector:   div#CybotCookiebotDialogBody
            > div.CybotCookiebotScrollContainer
            > div#CybotCookiebotDialogBodyContent
            > div#CybotCookiebotDialogBodyContentText
dimensions: 376×168 (≈63K sq px)
phases:     TTFB 453ms (8%) | Load Delay 0 | Load Time 0 | Render Delay 4972ms (92%)
```

The hero h1 (`I build / measurement / infrastructure.`) uses `clamp(44px, 13vw, 200px)` font-size and should render at ~175–200px per line × 3 lines = ~525–600px tall, well above Cookiebot's 168px. Yet LCP picks Cookiebot. `font-display: swap` is set on Instrument Serif (`src/lib/fonts.ts`) so the fallback Georgia should paint text at FCP (800ms) — no FOIT. So the hero *is* probably painting at FCP, just apparently not qualifying as LCP candidate (either it's being CSS-hidden initially by the client-component boundary, or there's a paint-timing edge LCP doesn't credit the h1 for).

**What this means for D1c:** The 5.4s home LCP is a first-visit synthetic number dominated by a third-party GDPR compliance tool loaded `beforeInteractive`. It is not representative of real-user LCP in the common returning-visitor case (Cookiebot cookie already set → no dialog). Optimizing "LCP as measured by localhost-Lighthouse-on-first-visit" conflates:
- Real-user first-visit experience (Cookiebot-dominated, matches this benchmark)
- Real-user returning-visitor experience (hero-dominated, which this benchmark can't measure)
- Synthetic localhost variance

The `WebVitalsReporter` (D1a) is the path to honest field LCP numbers once sGTM/BQ tagging catches up.

### Lever attempted: preconnect + dns-prefetch to third-party origins (null result)

Tried adding `<link rel="preconnect">` + `<link rel="dns-prefetch">` to `consent.cookiebot.com` and `www.googletagmanager.com` in `src/app/layout.tsx`'s `<head>`. Two consecutive runs: home LCP **6.4s** (both runs) vs 5.4s baseline. Perf 77–78 vs 80 baseline. Revert applied.

Explanation: on localhost, third-party network latency is effectively zero, so preconnect saves nothing. The extra connection warmup adds request contention during the critical rendering window, slightly slowing the critical path. In production (real latency + TLS), preconnect is typically a 100–300ms win per unique origin, but that gain is invisible to a local synthetic run. Committing the change would optimize an unmeasured-here case at a measurable-here cost.

**Carry forward:** If a Vercel-preview Lighthouse run is added to the D1b baseline flow, re-try preconnect there where the latency savings can be measured.

### Next lever candidates (mobile-specific, in order of expected measurable impact)

1. **Reduce font payload on mobile.** Four fonts total 243KB (`instrument-serif.ttf` 36KB + italic 36KB + `plus-jakarta-sans-var.ttf` 78KB + `jetbrains-mono-var.ttf` 93KB). On Slow 4G that's ~1.2s just for fonts. Levers: check if the italic Instrument Serif is used above the fold (if only used inside the overlay, it can be lazy-loaded); subset Jakarta + JetBrains to the actual glyph set; `size-adjust` / `ascent-override` to tighten the fallback-font metrics so FOUT doesn't cause a layout jank. Highest-confidence mobile win.
2. **Confirmation page LCP (mobile 8.2s).** Likely Metabase-iframe-related; the iframe's `src` triggers a cross-origin request that competes with the critical path under Slow-4G throttling. Skeleton-first paint, or lazy iframe load (`<iframe loading="lazy">` if it's below the fold) each worth trying.
3. **Bundle splits.** Baseline flagged five chunks >100KB on disk including a 476KB top chunk. Inspect via `@next/bundle-analyzer` or by reading chunk contents. Recharts on `/demo/ecommerce/analytics` is the obvious candidate for dynamic import.
4. **Hero h1 font-size tweak to restore LCP-winning status on mobile.** Small, reversible, symbolically useful even if it doesn't move render-delay. Gets F off the books.
5. **Mobile `/demo/ecommerce` trace_engine failure.** Investigate whether a Lighthouse config adjustment (e.g. skipping the after() fetch via a request-blocking list) captures a clean Perf score, or accept as tooling artifact.
6. **Preconnect + dns-prefetch** — re-attempt once a Vercel-preview Lighthouse run joins the flow, where real TLS latency makes preconnect a measurable win.

## Target CWV thresholds for Phase 10 D1

Google's 2026-current CWV thresholds (the web-vitals library applies these when it emits `metric_rating`):

| Metric | Good | Needs improvement | Poor |
|---|---|---|---|
| **LCP** (Largest Contentful Paint) | ≤ 2.5s | ≤ 4.0s | > 4.0s |
| **CLS** (Cumulative Layout Shift) | ≤ 0.1 | ≤ 0.25 | > 0.25 |
| **INP** (Interaction to Next Paint) | ≤ 200ms | ≤ 500ms | > 500ms |
| **FCP** (First Contentful Paint) | ≤ 1.8s | ≤ 3.0s | > 3.0s |
| **TTFB** (Time to First Byte) | ≤ 0.8s | ≤ 1.8s | > 1.8s |

**D1 completion target:** all three "Core" metrics (LCP, CLS, INP) rated `good` at the 75th percentile of real-user data on the three dynamic routes (`/`, `/demo/ecommerce`, `/demo/ecommerce/confirmation`). FCP + TTFB are secondary but reported.

## Measurement plan (how we gather numbers going forward)

**Dev-time:** The `WebVitalsReporter` in `src/components/scripts/web-vitals-reporter.tsx` pushes one `web_vital` event per metric per page visit onto `window.dataLayer`. In a dev console: `window.dataLayer.filter(e => e.event === 'web_vital')` returns the per-metric readout after page-hide / visibilitychange.

**Preview / production:** Once the sGTM web container picks up `web_vital` (Phase 11 D9 reconciler, or a small GTM update bundled with a D1 sub-commit), the events land in `iampatterson_raw.events_raw` and can be queried by `metric_name`, `metric_rating`, `navigation_type`, `page_path`.

**Lighthouse runs:** Automated by `scripts/capture-cwv-baseline.sh`. Spins up `npm run start` on a configurable port, runs Lighthouse against the three dynamic routes with `performance,accessibility,best-practices,seo` categories, writes JSON + HTML reports to `docs/perf/lighthouse-<date>-<route>.{json,html}`. Usage: `bash scripts/capture-cwv-baseline.sh`. Takes ~2-3 min end-to-end. A scripted Vercel-preview variant lands later if/when per-PR perf regression detection becomes load-bearing.
