# Load test — synthetic profile

**Phase:** 10d D6
**Date:** 2026-04-25
**Branch:** `phase/10d-launch-prep`
**Runner:** `scripts/run-load-tests.sh`

## Scope

Phase 10d D6 deliverable, verbatim:

> **Load testing.** Data generator + WebSocket service under synthetic
> load. Pub/Sub throughput profile. Document in
> `docs/perf/load-test-<date>.md`.

Two services are load-tested in-process via gated Jest suites:

1. **Event-stream service** (`infrastructure/cloud-run/event-stream`) —
   the Cloud Run SSE service that receives Pub/Sub push and fans out to
   browser EventSource connections.
2. **Data-generator service** (`infrastructure/cloud-run/data-generator`) —
   the Cloud Run service that generates synthetic events for the
   ecommerce / subscription / leadgen profiles, called by Cloud
   Scheduler on staggered daily cron.

Pub/Sub throughput is **not** load-tested live. Generating millions of
real Pub/Sub messages would burn GCP quota + cost without producing
signal beyond what the published Pub/Sub quotas already promise. The
throughput profile below is analytical, derived from Pub/Sub's
documented per-region quotas + the message size distribution measured
during D5 (security review).

## Methodology

### Why in-process load tests, not real network

Both services are stateless message processors. Their hot paths are:

- **Event-stream:** `parsePubSubMessage` (base64 decode + JSON parse +
  field validation) → `ConnectionManager.send` (Map lookup + write to
  Express Response). Both are pure CPU/memory work; the network layer
  (HTTPS termination, OIDC verification) is Cloud Run's, not the
  service's.
- **Data-generator:** `generateDateRange` (orchestrator) →
  per-business-model engine (CPU-bound event synthesis using a seeded
  RNG). Once the events are in memory, transport to sGTM is a separate
  outbound HTTP path that the existing `transport.test.ts` covers for
  correctness.

Stressing the service code in-process at high N produces the throughput
ceiling without the noise of network-layer variance. Network-layer
behaviour is best measured against a real Cloud Run preview instance,
which is captured below as a Phase 11 carry-forward.

### Gating — `LOAD_TEST=1`

Both load suites are gated on `LOAD_TEST=1` so they don't add minutes
to the default Jest runs. Mirrors the `E2E_ENABLED=1` pattern from the
Playwright suites.

### Determinism

The data-generator runs use `seed: 42` so re-runs are bit-identical.
The event-stream runs use deterministic synthetic payloads built in a
loop. Throughput numbers vary ~5-15% across runs on a developer laptop
due to GC timing and hardware temperature; the asserted floor is set
~50% below the median observed value to avoid flakes while still
catching real regressions.

## Results — committed baseline (2026-04-25)

Hardware: Apple M-series, 64 GB RAM, Node 20.9+. All times measured via
`process.hrtime.bigint()`; heap deltas via `process.memoryUsage()`.

### Event-stream service

| Profile | Ops | Duration | Throughput | Floor (asserted) |
| --- | ---: | ---: | ---: | ---: |
| `parsePubSubMessage` (parse-only) | 50,000 | 32.6 ms | **1,532,043 ops/sec** | 50,000 |
| Parse + route end-to-end (100k msgs × 5k sessions) | 100,000 | 132.3 ms | **756,058 ops/sec** | 25,000 |
| Reject malformed bodies | 50,000 | 73.3 ms | **681,775 ops/sec** | 100,000 |
| Broadcast to 1k connections | 1,000 | 0.1 ms | **8,281,573 ops/sec** | <100 ms |

**Read:** the in-process ceiling is well above any plausible real
traffic volume. At 5k connected sessions and 100k delivered messages
the routing path holds 756k ops/sec — three orders of magnitude above
the demo's expected ~10-100 events/sec sustained. The actual
production bottleneck is **not** in the service code; it's in the
upstream Pub/Sub push delivery rate (default ~1k push/sec/subscription
per region, see "Pub/Sub characteristics" below) and in Cloud Run's
concurrency settings.

### Data-generator service

| Profile | Sessions | Events | Duration | Throughput | Heap delta | Floor (asserted) |
| --- | ---: | ---: | ---: | ---: | ---: | ---: |
| `generateDay` × 5,000 sessions/day ecommerce | 3,750 | 14,193 | 67 ms | **212,763 events/sec** | +27.8 MB | 30,000 events/sec, ≥2,500 sessions |
| `generateDateRange` × 30 days at typical config (500/day) | 15,573 | 59,221 | 242 ms | **244,979 events/sec** | +111.8 MB | <30s, 10k-500k events |

**Read:** an 18-month backfill at typical demo volume (500 sessions/day
× ~547 days × ~3.8 events/session) projects to ~1.04M events generated
in ~4.2s. Production backfill is gated by sGTM transport throughput,
not generator throughput; the generator can produce events ~30× faster
than sGTM can ingest them. The 30-day run's heap delta of 111.8 MB
sets the upper bound on memory pressure for ongoing daily generation
(which only retains one day's events at a time, so peak working set is
much smaller).

**Determinism pin:** same seed + same config produces bit-identical
event count, session count, and event-name breakdown. This is what
makes the load profile a useful CI baseline — a regression in
generator behaviour shows up as a distribution shift, not noise.

## Pub/Sub throughput characteristics (analytical)

Phase 10d D6 specifies "Pub/Sub throughput profile" without specifying
load-against-real-Pub/Sub. The analytical profile below is derived
from Google's published quotas for the
`iampatterson-events` topic + the message-size distribution measured
during D5 (security review).

### Quotas in scope

Default per-region quotas applicable to a single Pub/Sub topic +
push subscription pair (us-central1):

| Quota | Limit | Real-world impact |
| --- | --- | --- |
| Topic publish throughput | 100 MB/s per region | Headroom is enormous for a demo workload. Single-message size in our payload is ~700 bytes (`stg_events` average), so 100 MB/s ≈ 150k msgs/sec. |
| Subscriber pull/push delivery | ~1k msgs/sec/subscription default | This is the actual production bottleneck for the SSE service. To go higher requires `--retain-acked-messages` + multiple subscriptions or higher subscriber concurrency. |
| Message size | 10 MB max | Our messages are ~700 bytes; 4 orders of magnitude under the cap. Not a constraint. |
| Retention duration | 7 days max (default), unacked | Long enough that a Cloud Run cold-start window (a few minutes) doesn't drop messages. |
| Acknowledgement deadline | 600s max | We ack inside the push handler before fanning to SSE listeners (see `event-stream/src/server.ts` POST `/pubsub/push`), so deadline isn't a constraint either. |

### Realistic demo workload

The demo's expected sustained traffic is ~10-100 events/sec at the
upper bound of "many simultaneous prospects browsing the site." That's
**3-4 orders of magnitude under the 1k msgs/sec push delivery
ceiling**. The synthetic generator emits at ~250k events/sec
in-process but routes through sGTM (rate-limited) before hitting
Pub/Sub, so the Pub/Sub path never sees a burst above sGTM's own
publish rate.

### Where this would actually break

Three failure modes worth naming, even though we're not at scale:

1. **Pub/Sub push subscription saturation.** If the SSE service
   consistently took >100ms to ack push messages (e.g. a Map lookup
   blocked on JSON parse of a 10MB payload), Pub/Sub would back off
   and queue. The 1k-msg/sec ceiling would actually bite. Mitigation
   is already in place: ack happens inside the synchronous handler
   before any fan-out work.
2. **EventSource connection cap.** Cloud Run's default concurrency is
   80 simultaneous requests per instance. SSE connections are
   long-lived requests, so 80 connected browsers per instance is
   the cap. Auto-scaling kicks in beyond that. Demo traffic at 10-100
   sessions/min stays well under one instance's headroom. Production
   monitoring (Phase 11 D3) should pin connection-count + instance
   count.
3. **ConnectionManager memory growth.** The Map grows linearly with
   active sessions; each entry holds an Express Response object plus
   a session_id string. At 5k connections (the load test profile),
   in-process memory stayed flat through the routing run (no GC
   pauses in the timed window). At 50k connections we'd exceed Cloud
   Run's default 512MB memory cap; that's a horizontal-scale concern,
   not a code-fix concern.

## Carry-forwards (Phase 11 D3 / D9)

Three observations the in-process load test can't measure that are
worth capturing for the operational-readiness sub-phase:

1. **Real Cloud Run preview load.** A `gcloud run deploy --no-traffic`
   preview of the event-stream service, hit with an Artillery or k6
   script that generates real HTTPS POSTs at 100-500 RPS, would
   measure HTTPS termination + OIDC verification overhead that the
   in-process test can't see. Phase 11 D3 (uptime checks +
   monitoring) is the natural home — the run's metrics also feed the
   Cloud Run dashboard alerts.

2. **sGTM transport rate-limit profile.** The data-generator's
   `transport.ts` calls sGTM via `/g/collect`. sGTM's per-container
   ingest rate isn't published; in production we observe ~50-100
   req/sec sustained without errors. Worth measuring against a
   non-prod sGTM container before the next backfill to know whether
   bursts >500/sec produce 4xx/5xx responses.

3. **Browser-side `useEventStream` reconnect under sustained server
   load.** D2 pinned the buffer-survives-reconnect path; this load
   test pins the server-side throughput. The end-to-end "1k browser
   tabs simultaneously reconnecting after a Cloud Run rolling restart"
   scenario isn't covered by either, and would surface any
   thundering-herd issue that the D5 jitter doesn't fully absorb.
   Phase 11 D3 chaos-test scope.

## Reproducing the baseline

```bash
# From repo root
./scripts/run-load-tests.sh
```

The script:
1. `npm install`s service deps inside both Cloud Run service dirs if
   `node_modules` is missing.
2. Runs each load suite with `LOAD_TEST=1`.
3. Prints per-test throughput numbers via `console.log` (visible in
   the Jest output).

Compare the printed numbers against the table above. Throughput should
hold within ~15% of the committed baseline; a sustained drop below the
asserted floor is a regression worth investigating.

## Summary

| Service | Real-world ceiling | Headroom vs demo workload |
| --- | --- | --- |
| Event-stream parse | 1.5M msgs/sec | 5+ orders of magnitude |
| Event-stream end-to-end routing | 750k msgs/sec | 4-5 orders of magnitude |
| Data-generator events/sec | 240k events/sec | 3+ orders of magnitude vs daily backfill |
| Pub/Sub push delivery | 1k msgs/sec (per subscription) | 1-2 orders of magnitude — this is the production-facing ceiling |

**Result:** the demo's worst-case workload sits 1-2 orders of
magnitude below the lowest production ceiling (Pub/Sub push). The
service code is not a bottleneck. Phase 11 D3 owns the operational
monitoring that detects approach-to-ceiling at runtime; D6 pins the
in-process baselines against which future regressions can be measured.
