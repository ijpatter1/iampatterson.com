### CRT boot sequence restoration (commit `4e185f7`)

**What:** UAT flagged that the under-the-hood overlay's boot effect wasn't playing, visitors saw a persistent amber glow across the top nav but no "terminal powering on" moment. Audit against `docs/input_artifacts/iampatterson-com/project/iampatterson-interactive/styles/overlay.css` confirmed three divergences from the prototype: (1) our `.crt-bloom` was a persistent amber radial (the glow the user was seeing) instead of the prototype's black paint-down curtain; (2) `.crt-flicker` looped forever instead of firing once during boot; (3) `.tab-flash` was a gentle 420ms ease-out fade instead of the prototype's hard `steps(3, end)` strobe.

**Fix:** Preserved the amber glow the user liked by renaming the old layer to `.crt-ambient`. Ported the prototype's four-layer sequence:
- `.crt-ambient`, persistent warm radial (what the user specifically asked to keep)
- `.crt-bloom`, opaque black curtain, paints down via `crt-paint-down 720ms steps(14, end) both` during `[data-phase='boot']`
- `.crt-flicker`, one-shot amber pulse via `crt-flick 420ms steps(8, end) both` during `[data-phase='boot']`
- `.crt-scanlines`, fades in on `[data-phase='on']`
- `.tab-flash`, hard strobe (`150ms steps(3, end)` with opacity 0 → 0.9 → 0.2 → 1), hides panel contents during boot

CRT field now mounts during both `phase === 'boot'` and `phase === 'on'` (previously `'on'`-only) so the boot animations have a target to animate.

**Notable implementation decisions:**
- Kept the amber glow per user's explicit preservation request ("I do quite like this effect though, so take note of how you've achieved it")
- Used `[data-phase]` attribute selectors to scope boot-only animations, matching the existing `data-phase={phase}` attribute already on the outer container
- Numeric fidelity to the prototype maintained: 720/420/150ms durations, step counts, keyframe opacities all match

**Tests added/changed:** 1 assertion flip in `tests/unit/components/overlay/under-the-hood-view.test.tsx`, CRT field is now expected in the DOM during boot (required for the paint-down curtain to animate). No new tests written for the new `.crt-ambient` layer or the boot-phase `.tab-flash` opacity: 0 behavior (flagged as test-coverage gap by both reviewers, see Issues below).

## Issues & Technical Debt

Both reviewers flagged the same two Minor/Important issues. User accepted UAT with these unresolved; capturing here for follow-up.

1. **Paint-down curtain is effectively invisible and snaps on phase transition**, Important (evaluator), Minor (product reviewer). The `.crt-bloom` black curtain animates for 720ms but the JS boot hold is only 260ms, so the paint-down reaches ~36% before `data-phase` flips to `'on'` and the animation is torn down (no `[data-phase='on'] .crt-bloom` rule to fade it out). Additionally, the curtain is `#0d0b0a`, the same color as the panel it paints over, so the motion is essentially not visible to the user. The prototype handles this with `.overlay-host.phase-on .crt-bloom { opacity: 0; clip-path: inset(0 0 0% 0); transition: opacity 200ms ease 80ms; }`, a graceful fade-out that was not ported. Resolution options: (a) port the phase-on fade rule + lengthen `BOOT_DURATION_MS` to let the paint-down complete, or (b) drop the `.crt-bloom` layer entirely since it's currently a no-op. Location: `src/styles/globals.css:75-96`, `src/components/overlay/under-the-hood-view.tsx:21`.

2. **Tab buttons and body are `position: relative`, CRT field doesn't cover them**, Important (evaluator). The CRT field is `absolute inset-0` with no explicit z-index. Tabs (`src/components/overlay/under-the-hood-view.tsx:45`, `className="relative ..."`) and body (`:202`, `className="relative flex-1 ..."`) are positioned, so they paint ON TOP of the CRT field. The amber-across-the-top-nav reading works because the header is in-flow non-positioned, but the flicker/scanlines don't blend over the body region or through the tab button text. Prototype gives `.crt-field` an explicit `z-index: 3` above the panel's `z-index: 2` (`overlay.css:46, 119`), which wasn't carried over. Resolution: add `z-index` to the CRT field wrapper. Location: `src/components/overlay/under-the-hood-view.tsx:158`.

3. **No tests verify the new animation wiring**, Important (evaluator), Minor (product reviewer). The only test change was flipping a single assertion. No test asserts the `.crt-ambient` layer renders, that `.tab-flash` receives `opacity: 0` during boot, or that all four CRT layers are present. Tests were updated after the fix, not red-first. Given this was already a visual regression that shipped undetected, a tighter test baseline would reduce re-regression risk. Location: `tests/unit/components/overlay/under-the-hood-view.test.tsx`.

4. **Flicker layer ordered above bloom, inverting the prototype**, Minor. Prototype DOM order is flicker → bloom → scanlines (`overlay.jsx:229-231`); implementation is ambient → bloom → flicker → scanlines. The warm flicker lands on top of the already-painted black curtain rather than being occluded by it. Subtle visual difference, unlikely to be noticed. Location: `src/components/overlay/under-the-hood-view.tsx:160-163`.

5. **Hardcoded `#0d0b0a` in `.crt-bloom`**, Minor (evaluator). Could reference a CSS custom property mirroring `u-paper` from the Tailwind config. Location: `src/styles/globals.css:77`.

## Session Notes

- **Stacking-context realization:** Documenting this for future context, the outer overlay container is `fixed z-50` creating a new stacking context. Inside, positioned elements with `z:auto` paint ABOVE in-flow flex children. So the CRT field (absolute, z:auto) sits above header/tabs/body wrappers, which is what allows `mix-blend-mode: screen` on the ambient layer to blend across the top nav. This is load-bearing for the effect; don't break it by giving header/tabs `position: absolute` or positive z-index without also explicitly z-ordering the CRT field.
- **Amber-glow preservation pattern:** When a user says "I like this effect, note how you've achieved it," the right move is to name and preserve it as a standalone layer rather than letting it get entangled with other changes. `.crt-ambient` is that layer.
- **Prototype fidelity vs. harness timing:** The prototype sets `BOOT_DURATION_MS = 260` but the curtain animation is `720ms`. Mismatches like this need a phase-on handoff rule or they snap. Worth auditing other animation durations against phase hold times in future overlay work.
