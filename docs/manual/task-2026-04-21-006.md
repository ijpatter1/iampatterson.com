## Context

UAT r1 item 15 read "The dashboard isn't loading" on the `/demo/ecommerce/confirmation` page. The confirmation page embeds one full Metabase dashboard via a server-signed JWT. There are three paths where "isn't loading" can happen; which one you see determines the fix.

The signing path is at `src/app/demo/ecommerce/confirmation/page.tsx`:

- If `METABASE_EMBED_CONFIG` or `MB_EMBEDDING_SECRET_KEY` env vars are missing, `mintConfirmationDashboardUrl` returns `null` and `DashboardPayoff` renders the "dashboard embeds disabled" fallback with a deep-link to `bi.iampatterson.com/dashboard/2` behind Google SSO.
- If both env vars are set and valid, the page renders an `<iframe src={signedUrl}>`. The iframe's load outcome depends on Metabase reachability + warm state.

The dual-evaluator fix pack (commit `7c79129`) already confirmed the signer code path is correct; the behaviour depends entirely on deploy-time env + Metabase state.

## What you're seeing, figure out which path

Visit `/demo/ecommerce/confirmation` on the environment where you hit this (likely a local `npm run dev` or a Vercel preview) and open the browser devtools.

### Path A, "dashboard embeds disabled in this environment"

If the page shows an amber `TIER 3 · DASHBOARDS` panel with the text "the signing env vars aren't wired in this environment," **the env vars are missing**. This is the code's honest fallback.

**Fix:** set both in the relevant environment:
- `MB_EMBEDDING_SECRET_KEY`, mirror of Secret Manager `metabase-embedding-secret-key`
- `METABASE_EMBED_CONFIG`, mirror of Secret Manager `metabase-embed-config`

For Vercel: `vercel env add <name>` or set via the dashboard. For local `npm run dev`: create `.env.local` (gitignored) with both values.

After setting, restart `npm run dev` or trigger a redeploy.

### Path B, iframe renders but shows "This question couldn't be embedded"

The JWT is either malformed, signed with the wrong secret, or the dashboard id in `METABASE_EMBED_CONFIG.dashboardId` doesn't match what's published-embeddable on the Metabase instance.

**Check:**
1. `gcloud secrets versions access latest --secret=metabase-embed-config --project=iampatterson`, note the `dashboardId` field.
2. On Metabase at `bi.iampatterson.com`, open that dashboard and confirm "Embedding" is ON and "Enable embedding" is saved.
3. Confirm `MB_EMBEDDING_SECRET_KEY` matches Metabase's "Embedding secret key" (Admin → Settings → Embedding).

### Path C, iframe spins indefinitely

Metabase cold-start. Cloud Run JVM warmup after idle. The session handoff `docs/sessions/session-2026-04-21-002.md` notes `--no-cpu-throttling` was applied earlier, but verify:

```
gcloud run services describe metabase --region=us-central1 --project=iampatterson --format='value(spec.template.spec.template.metadata.annotations."run.googleapis.com/cpu-throttling")'
```

Expected: `false` (or unset if `--no-cpu-throttling` is the default). If it returns `true`, re-run:

```
gcloud run services update metabase --region=us-central1 --project=iampatterson --no-cpu-throttling
```

Also check `minScale`:

```
gcloud run services describe metabase --region=us-central1 --project=iampatterson --format='value(spec.template.metadata.annotations."autoscaling.knative.dev/minScale")'
```

Expected: `1` (keeps at least one warm instance). If `0`, bump to `1`:

```
gcloud run services update metabase --region=us-central1 --project=iampatterson --min-instances=1
```

### Path D, iframe renders but body is blank white

The iframe loads the embed HTML shell (200 OK) but shows nothing. In devtools Network, all requests to `/app/*` on `bi.iampatterson.com` show `302 → accounts.google.com/o/oauth2/v2/auth` with header `x-goog-iap-generated-response: true`.

**Root cause:** the LB URL-map pathMatcher routes `/api/*` and `/embed/*` to the non-IAP backend, but not `/app/*` (Metabase's JS/CSS bundle path). Asset requests fall through to the IAP-gated backend and IAP rewrites them to its login page, which cannot render inside an iframe (CSP `frame-ancestors`).

**Fix (applied 2026-04-22):** add `/app/*` to the pathMatcher:

```
gcloud compute url-maps export metabase-url-map --destination=/tmp/metabase-url-map.yaml --project=iampatterson
# add "- /app/*" to pathMatchers[0].pathRules[0].paths (same indent as /api/*, /embed/*)
gcloud compute url-maps import metabase-url-map --source=/tmp/metabase-url-map.yaml --project=iampatterson
```

Defensive future additions if new asset families surface in a HAR: `/public/*`, `/fonts/*`, `/favicon.ico`.

### Path E, assets load but cold-start exceeds ship-gate thresholds

Incognito load, all assets return 200, dashboard renders, but first paint >10s (30–45s observed 2026-04-22).

**Root cause:** not JVM cold-start (`minScale=1` prevents that); this is Metabase application warmup, per-card BigQuery queries cold-miss Metabase's card cache and BigQuery's 24h query cache, serialising ~3 card queries at 5–15s each.

**Fix:** the three-layer stack in Phase 9F deliverable 9, Cloud Run config (already applied), Metabase dashboard caching (TTL 24h on dashboard id 2, set once in admin), and runtime organic warmup (`src/lib/metabase/keep-warm.ts` fired from `/` and `/demo/ecommerce`).

After the full stack ships, re-run the ship-gate probe below from a fresh-deploy state (resets the warmup module's 30-min debounce so the next request *is* a cold-path probe).
