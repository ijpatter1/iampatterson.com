### Notable implementation decisions

- **Inline embeds on the confirmation page, not overlay.** Per user direction: the overlay is the *Tier 2 plumbing* surface at each funnel step; the confirmation page is the *Tier 3 payoff* surface. Inline iframes frame the visitor's specific purchase against aggregate data in a way the overlay can't. Overlay gets the three non-embeddable questions as summary cards so it stays useful on the confirmation route without duplicating.
- **Server Component signer, not API route.** Signing at render time avoids a client-side fetch round-trip and a flash of mocks → embeds. 10-minute TTL is long enough for a tab-switch, short enough that leaked URLs have small shelf life.
- **Single JSON env var `METABASE_EMBED_CONFIG` instead of three separate vars.** Zero-translation mirror of the Secret Manager entry produced by `apply.sh --publish-embed-config`. One rotation target, not four.
- **Real `onLoad`-driven opacity transition for loading state.** The Pass 1 CSS-only version was inert (iframe paints opaquely on mount regardless of content load). The fix extracted `LiveEmbedFrame` as a client component with `useState(loaded)` + `onLoad` toggling both iframe and placeholder opacity via 500ms transitions.
- **Path-agnostic copy hedging on the Tier 3 surface.** Visitors arrive via two paths (organic checkout funnel → `purchase` event fires, OR Services cross-link → no purchase event). Copy that assumes "your event just fired this session" overclaims on path 2. Hedged to "orders like yours" + "the checkout funnel fires…" (present-tense infrastructure-descriptive) so captions hold on both paths.
- **Non-embeddable card IDs hardcoded in `dashboard-view.tsx` with a sync comment.** The architectural fix (server→client env plumbing or `NEXT_PUBLIC_` env var) was deemed out of proportion to the risk. IDs are stable across `apply.sh` re-runs via name-based idempotency; comment points at `.ids.json` as the sync source.
