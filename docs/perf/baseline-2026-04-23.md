**Readout:**
- **LCP is the D1c optimization target.** Home 5.4s and confirmation 8.2s both land deep in the "poor" bucket (>4.0s). The CWV thresholds call for ≤2.5s to rate "good" at p75. Every other metric is already comfortably green on home + confirmation.
- **CLS is already excellent** across all three routes (0.000–0.001). No D1c work needed there.
- **INP is not a Lighthouse audit** (it's a field-only metric). The `WebVitalsReporter` landed in D1a is still how real INP numbers will land — Lighthouse reports `max-potential-fid` (70–80ms across routes) as the closest synthetic proxy, and those are in the "good" band.
- **`/demo/ecommerce` perf score shows as 0** due to a non-fatal `@paulirish/trace_engine` insight-computation error during the Lighthouse run. Likely the `after()` hook's background fetch to `bi.iampatterson.com` tripping the synthetic trace engine (the existing Phase 10a smoke test notes the same behaviour in a different form: `waitUntil: 'networkidle'` hangs on the warmup's network chain). FCP + CLS + Speed Index still captured cleanly; LCP/TBT/TTI are undefined in the audit output. This is a synthetic-tooling quirk, not a real-user problem — real users don't run Lighthouse.
- **Confirmation's BP score (79, vs 100 on the other routes)** is worth a look during D1c. The `.report.html` under `docs/perf/` surfaces the specific audits that regressed.

**Hypothesis for LCP:** the hero's serif h1 (Instrument Serif font) plus any hero-adjacent images are likely candidates. Fonts preload cleanly per the response headers, but the LCP element's paint may be delayed by the ambient-bubbles canvas, the pipeline section's CSS-only work, or the overlay provider's boot tree. D1c opens by profiling the homepage LCP element specifically.

## Target CWV thresholds for Phase 10 D1

Google's 2026-current CWV thresholds (the web-vitals library applies these when it emits `metric_rating`):

| Metric | Good | Needs improvement | Poor |
|---|---|---|---|
| **LCP** (Largest Contentful Paint) | ≤ 2.5s | ≤ 4.0s | > 4.0s |
| **CLS** (Cumulative Layout Shift) | ≤ 0.1 | ≤ 0.25 | > 0.25 |
| **INP** (Interaction to Next Paint) | ≤ 200ms | ≤ 500ms | > 500ms |
| **FCP** (First Contentful Paint) | ≤ 1.8s | ≤ 3.0s | > 3.0s |
| **TTFB** (Time to First Byte) | ≤ 0.8s | ≤ 1.8s | > 1.8s |

**D1 completion target:** all three "Core" metrics (LCP, CLS, INP) rated `good` at the 75th percentile of real-user data on the three dynamic routes (`/`, `/demo/ecommerce`, `/demo/ecommerce/confirmation`). FCP + TTFB are secondary but reported.

## Measurement plan (how we gather numbers going forward)

**Dev-time:** The `WebVitalsReporter` in `src/components/scripts/web-vitals-reporter.tsx` pushes one `web_vital` event per metric per page visit onto `window.dataLayer`. In a dev console: `window.dataLayer.filter(e => e.event === 'web_vital')` returns the per-metric readout after page-hide / visibilitychange.

**Preview / production:** Once the sGTM web container picks up `web_vital` (Phase 11 D9 reconciler, or a small GTM update bundled with a D1 sub-commit), the events land in `iampatterson_raw.events_raw` and can be queried by `metric_name`, `metric_rating`, `navigation_type`, `page_path`.

**Lighthouse runs:** Automated by `scripts/capture-cwv-baseline.sh`. Spins up `npm run start` on a configurable port, runs Lighthouse against the three dynamic routes with `performance,accessibility,best-practices,seo` categories, writes JSON + HTML reports to `docs/perf/lighthouse-<date>-<route>.{json,html}`. Usage: `bash scripts/capture-cwv-baseline.sh`. Takes ~2-3 min end-to-end. A scripted Vercel-preview variant lands later if/when per-PR perf regression detection becomes load-bearing.
