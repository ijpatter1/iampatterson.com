## Session Notes

### Pattern established: `userOverride ?? consentDefault` for derived-default-until-user-intent state

D8's checked-state derivation generalizes: when a component needs to follow a derived default (from async-populated context) until the user explicitly interacts, store `userOverride: boolean | null` state and derive the actual value every render via `userOverride ?? derivedDefault`. `null` encodes "has not interacted, default governs"; a `boolean` encodes "user clicked, their intent pins." This pattern closes the `useState(initialFromAsyncContext)` anti-pattern (initial value captured from a still-null context; useState doesn't re-seed on subsequent renders). Applies anywhere else in the app where a component might be tempted to do `useState(state?.foo === 'bar')`, see D8 component JSDoc for the full rationale + file:line reference to the provider's async init effect.

### Pattern established: `jest.useFakeTimers({ doNotFake: [...] })` for rAF/cAF spy tests

D5's cleanup-on-unmount test had to reorder `jest.useFakeTimers()` first and opt out of faking `requestAnimationFrame` and `cancelAnimationFrame` so the per-test spies aren't clobbered by Jest's internal fake-timer machinery. Modern Jest (non-legacy fake timers) replaces rAF/cAF with its own implementations when `useFakeTimers()` is called; opting them out via `doNotFake` is the principled escape hatch. Applies to any test that needs to assert on rAF/cAF call patterns (memory leak detection, cleanup verification).

### Pattern established: real-identifier pinning against infrastructure JSON

D5's content schema (`src/lib/content/pipeline.ts`) pins the real GTM container IDs (`GTM-MWHFMTZN` web, `GTM-NTTKZFWD` server) against `infrastructure/gtm/*.json` via tests. The Pass 1 Critical finding (a fabricated `GTM-N5JJ7VT4` rendered as production reality with a docstring claiming it was real) would have slipped past review without this pinning. Pattern: any schematic / visualization / portfolio surface that displays production identifiers should pin them in tests against the authoritative infrastructure source, with the failure-mode inline-commented so future maintainers know what they're giving up if they delete the pinning.
