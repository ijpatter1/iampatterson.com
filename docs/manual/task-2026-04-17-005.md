#### Recovery: backend heals via delete+recreate, but can't delete while URL map references it

If `setup-domain.sh` reports that `metabase-backend` has a `portName` incompatible with serverless NEGs (e.g. `portName='http'` or `'https'`), its heal-in-place path tries to delete and recreate the backend. That delete fails when the URL map still references `metabase-backend` as its `defaultService`:

```
ERROR: (gcloud.compute.backend-services.delete) Some requests did not succeed:
 - The backend_service resource '...backendServices/metabase-backend' is
   already being used by '...urlMaps/metabase-url-map'
```

The script also refuses to delete while IAP is attached to the backend (silent IAP-config wipe would require re-running `setup-iap.sh` anyway). Full recovery:

```bash
# 1. Disable IAP on metabase-backend so the heal can proceed
gcloud iap web disable --resource-type=backend-services \
  --service=metabase-backend --project=iampatterson

# 2. Create metabase-backend-direct ahead of schedule (same config
#    setup-domain.sh's step 8 would create later)
gcloud compute backend-services create metabase-backend-direct \
  --global --load-balancing-scheme=EXTERNAL_MANAGED --project=iampatterson
gcloud compute backend-services add-backend metabase-backend-direct \
  --global --network-endpoint-group=metabase-neg \
  --network-endpoint-group-region=us-central1 --project=iampatterson

# 3. Swap the URL map's default-service away from metabase-backend
gcloud compute url-maps set-default-service metabase-url-map \
  --default-service=metabase-backend-direct \
  --global --project=iampatterson

# 4. Delete the misconfigured backend (now unreferenced)
gcloud compute backend-services delete metabase-backend \
  --global --project=iampatterson --quiet

# 5. Re-run setup-domain.sh — step 4 creates a clean metabase-backend
#    without portName, attaches the NEG; step 8 adds the path matcher
#    (--new-hosts='*' routes all traffic through it, so the URL map's
#    top-level default is effectively unreachable afterwards)
./setup-domain.sh --no-wait

# 6. Re-enable IAP on the new backend (idempotent: reuses the existing
#    OAuth client by display name, re-adds the allowlist)
./setup-iap.sh
```

**Exposure window** during steps 3-5: `bi.iampatterson.com` is fronted by a backend with no IAP. Metabase's own login still gates the UI, so the practical exposure is "random visitor hits the Metabase login page". Low risk; keep the window short by running steps 2-6 back-to-back.

**Survives the heal:** static IP, SSL cert (still `ACTIVE`), HTTPS proxy, forwarding rule, URL map, NEG, OAuth client + secrets, IAP allowlist. Only `metabase-backend` gets recreated.

### 2. Enable static embedding + generate an admin API key in Metabase

Both are one-time admin toggles. Do static embedding **first** — without it, Step 6's `apply.sh` gets an HTTP 400 "Embedding is not enabled." on the first `PUT /api/card/:id` that carries `enable_embedding: true` (the Funnel question is the first card in the spec order, so that's where it surfaces).

**2a. Enable static embedding.**

1. Open `https://bi.iampatterson.com/admin/settings/embedding-in-other-applications` (log in via IAP if prompted).
2. Under **Static embedding**, click **Enable**.
3. Copy the auto-generated **Embedding secret key** somewhere safe — this is `MB_EMBEDDING_SECRET_KEY` and deliverable 6b's Next.js signer will use it to mint JWTs. Treat it as any other secret (don't commit, don't paste into issues, etc.).

**2b. Generate an admin API key.**

1. Open `https://bi.iampatterson.com/admin/settings/authentication/api-keys`.
2. Click **Create API Key**.
3. **Key Name:** `apply-sh` (or similar — purely descriptive).
4. **Group:** `Admin`.
5. Click **Create**. Copy the key value — it is shown ONCE.

### 3. Store the key in Secret Manager

```bash
# Replace <KEY> with the value you just copied
echo -n "<KEY>" | gcloud secrets create metabase-api-key \
  --project=iampatterson \
  --replication-policy="automatic" \
  --data-file=-
```

Grant your own account access to read it (so `apply.sh` can fetch it):

```bash
gcloud secrets add-iam-policy-binding metabase-api-key \
  --project=iampatterson \
  --member="user:$(gcloud config get-value account)" \
  --role="roles/secretmanager.secretAccessor"
```

### Known gotchas surfaced on the 2026-04-18 live apply

Preserved here so a cold rebuild of this task doesn't re-discover them:

1. **gcloud auth to the wrong project** presents as "Cloud Run service 'metabase' not found in us-central1" at the Step 1 precondition. Covered in Preflight.
2. **`metabase-backend` accumulated `portName='http'`** somewhere between Phase 9B-infra Task 5 completion and this apply. Setup-domain.sh's heal-in-place path assumes the backend can be deleted, but the URL map's `defaultService` reference and IAP attachment both block that. The full recovery dance is in the "Recovery" subsection under Step 1.
3. **Static embedding is a separate admin toggle** from per-card `enable_embedding: true`. README flagged it; this task card didn't until now. Covered in Step 2a.
4. **`PUT /api/dashboard/:id` validates dashcard id uniqueness** in Metabase v0.59+, rejecting the legacy `id: -1` pattern. Fixed in apply.sh (commit `7d8a683`, `fix(metabase): give new dashcards unique negative ids for v0.59+ apply`). Preflight verifies the fix is in place.
5. **`/embed/question/:jwt` returns 200 + HTML on bad JWT**, not 4xx. Client-side JS validates the token. Both `setup-domain.sh`'s trailing echo and this task card's Step 1 verify had the wrong expected status — corrected here.

## Rollback

If something goes wrong:

- **Delete a stray card:** Metabase UI → card → three-dot menu → Archive.
- **Delete the entire collection:** Metabase UI → E-Commerce Dashboards collection → three-dot menu → Archive. On next `apply.sh` run, the script recreates everything from the specs.
- **Revoke the API key:** Metabase UI → Admin → Authentication → API Keys → delete. Then delete the Secret Manager secret.
- **Remove the URL-map split:** `gcloud compute url-maps remove-path-matcher metabase-url-map --path-matcher-name=direct-paths --global --project=iampatterson`, then `gcloud compute backend-services delete metabase-backend-direct --global --project=iampatterson`. `setup-domain.sh` re-adds both on next run.
