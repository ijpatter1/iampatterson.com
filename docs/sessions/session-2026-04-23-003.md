### What the evaluator loop actually caught

Items that would NOT have shipped without the 3-pass review cycle:

1. **SessionPulse stayed on the pre-10a idiom** despite the original "all 31 cleared" claim. `src/components/chrome/session-pulse.tsx` rendered next to `LiveStrip` and they'd have resolved session identity through two different mechanisms after the original sweep. Caught in Pass-2, fixed in `15fdda1`.
2. **`useSessionId.getSnapshot` was impure** — `getSessionId()` wrote `document.cookie` on every React concurrent-mode consistency check. Would have caused cookie thrash in prod. Caught in Pass-1, fixed in `6bcd2de`.
3. **`notifySessionCookieChange` was self-referential** — exported but no production writer called it. Caught in Pass-2, fixed by moving the listener set to `src/lib/events/session.ts` alongside `setSessionCookie` (the actual writer).
4. **Latent `orderId` regen bug in `checkout-form.tsx`** — the pre-10a `useMemo([total, itemCount], () => Date.now())` would re-mint the ID on cart edits mid-checkout, breaking the `begin_checkout` → confirmation `order_id=` join in BigQuery. Surfaced by the `react-hooks/purity` rule; fixed with lazy `useState` init + regression pin.
5. **`useSessionContext` had a no-op `subscribeSessionCookie`** — pre-existing asymmetry where `useSessionId` consumers saw cookie rotations on the current render tick while `useSessionContext` consumers only saw them on the 5s tick. Caught in Pass-3, unified on the shared subscription channel.
6. **`setSessionCookie` fired notify on same-value writes** — `getSessionId`'s max-age refresh path called `setSessionCookie` repeatedly with the same ID, firing ~20 no-op notifies per 20-event session. Bounded by `useSyncExternalStore`'s Object.is bailout but semantically wrong. Pass-3 surfaced; fixed at the source with a before/after comparison.

### Pattern: evaluator loop catches the author's own blind spots

The original preflight refactor's commit message claimed "all 31 violations cleared." Pass-2 of the evaluator loop revealed `SessionPulse` had been missed — the claim was only incidentally true because the `react-hooks/set-state-in-effect` rule flags first-setState-per-effect, so the missing migration didn't surface as a warning after the other effect-level disables were in place. This is exactly the kind of "blind spot the author can't see because they wrote the code" that an independent review loop is for. The Pass-2 fix-pack commit message calls out the original claim as inaccurate — self-correcting honesty rather than covering up.

### Pattern: drift-resistant doc phrasing

Pass-3 Tech Minor #1 surfaced that my own fix-pack swept 1168→1180 but introduced 1181 drift at its own accounting line (the +1 SSR test). Rather than chase the exact number forever, the Pass-3 sweep replaced specific counts with "`npm test` for current count" in the banner + D4/D5 descriptions. This breaks the class of drift, not just the instance. `git log main..HEAD` is the authoritative commit view; `npm test` is the authoritative test count. Specific numbers are point-in-time snapshots.

### Pattern: listener set belongs with the writer, not the reader

The Pass-1 fix-pack put `notifySessionCookieChange` inside `src/hooks/useSessionId.ts`. Pass-2 caught that this is architecturally wrong — the listener set should live next to the writer (`setSessionCookie` in `src/lib/events/session.ts`) so every cookie-write path participates in the notify channel automatically. Moving it also simplified the hook module (no more self-referential notify export; it's just a re-export for tests + any future non-setSessionCookie writer). The right layer for a pub/sub channel is the layer that OWNS the subject, not any particular consumer.
