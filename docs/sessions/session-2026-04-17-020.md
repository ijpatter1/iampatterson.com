### Task 3, Cloud Run service deployment (✅ live)

**Wrapper script + 2 fixes across 3 commits.**

- `docs/manual/task-2026-04-17-001.sh` (`cdbaaf5`), self-contained wrapper that auto-resolves the latest `v0.59.6.x` tag from GitHub releases API (paginated), runs dry-run preview, prompts for confirmation, deploys, then runs 5 verification checks (Ready=True, image match, URL assigned, ingress locked, `.run.app` direct access blocked).
- `3c90c8e` fix: `deploy.sh` leak guard matched `${VAR}` in its own header comment. Strip comment lines before matching.
- `7b4708e` fix: architectural switch from Cloud SQL Auth Proxy Unix socket to **TCP over Cloud SQL private IP** via the existing VPC peering. Metabase's bundled pgjdbc concatenates `:PORT` onto `MB_DB_HOST`, turning `/cloudsql/<instance>` into `jdbc:postgresql:///cloudsql/...:5432/metabase`, malformed. Cloud SQL JDBC Socket Factory isn't bundled. TCP private IP is functionally equivalent security (traffic never leaves the VPC). Same commit adds `MB_JETTY_PORT=8080` because Metabase's Jetty defaults to 3000 and doesn't auto-read Cloud Run's injected `PORT`. `deploy.sh` resolves the private IP at render time via `gcloud sql instances describe | jq` with a PRIVATE-type filter.
- `a237c5b` PHASE_STATUS: Task 3 → ✅, deployed image `metabase/metabase:v0.59.6`, Cloud Run URL `https://metabase-eb4xrwmo3q-uc.a.run.app`.

### Task 5, External HTTPS load balancer + cert (✅ live)

**Two fixes across 3 commits.**

- `93d5277` fix: backend service for serverless NEG rejects any `portName`. `setup-domain.sh` had `--protocol=HTTPS` on create, which auto-sets `portName=https`. Removed `--protocol` and added heal-in-place: if an existing backend has a non-empty portName, delete and recreate. `ea54a6d` (post-review) added an IAP-state precheck so the destructive heal never silently destroys IAP config.
- `a77e431` fix: LB-via-serverless-NEG needs `allUsers run.invoker` on the Cloud Run service. Cloud Run's default "require authentication" gate refuses anonymous requests from the LB; serverless NEGs don't carry a service identity pre-IAP. Added idempotent binding to `deploy.sh` with a multi-paragraph comment explaining it's the Google-documented pattern and ingress lock + IAP are the real security boundaries. `dc1d727` PHASE_STATUS: Task 5 → ✅ with static IP `34.102.206.180`, SSL cert ACTIVE, all 7 components provisioned.

### Task 6, IAP configuration (✅ enforcing)

**Two fixes across 3 commits.**

- `8366674` fix: `setup-iap.sh` silently exited on a fresh project because `iap.googleapis.com` wasn't enabled and stderr was routed to `/dev/null`. Auto-enable the API up front, then use a tempfile for brand-list output so real gcloud errors reach the user. Added the deprecation notice for the IAP OAuth Admin APIs (March 19, 2026 sunset for new projects; grandfathered for existing brands).
- `927e8db` fix: once IAP is enforcing, requests reach Cloud Run as the IAP service agent, not anonymously. The `allUsers` binding isn't sufficient, the IAP service agent needs its own `run.invoker`. Symptom: "The IAP service account is not provisioned" on the browser page. Added two idempotent steps to `setup-iap.sh`: `gcloud beta services identity create --service=iap.googleapis.com` (provisions the agent) and `gcloud run services add-iam-policy-binding` with the agent's SA email (`service-<PROJECT_NUMBER>@gcp-sa-iap.iam.gserviceaccount.com`). `77a9bb6` PHASE_STATUS: Task 6 → ✅, `curl -sI https://bi.iampatterson.com/` returns HTTP/2 302 to `accounts.google.com` with the `GCP_IAP_XSRF_NONCE` cookie set.
