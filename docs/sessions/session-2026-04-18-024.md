### Pass 1: CRT boot sequence restoration (`4e185f7`)

**What:** UAT + screenshot from user showed a persistent amber glow across the top nav with no "terminal powering on" gesture. Audit against the editorial prototype at `docs/input_artifacts/iampatterson-com/project/iampatterson-interactive/styles/overlay.css` identified three divergences:
1. `.crt-bloom` was a persistent amber radial (the glow the user was seeing) instead of the prototype's black paint-down curtain
2. `.crt-flicker` looped forever via `crt-pulse 3.2s ease-in-out infinite` instead of firing once during boot
3. `.tab-flash` was a gentle 420ms ease-out with translate instead of the prototype's hard `150ms steps(3, end)` strobe

**Fix:** Preserved the amber glow at the user's explicit request ("I do quite like this effect though, so take note of how you've achieved it") by renaming the old `.crt-bloom` → `.crt-ambient`. Ported the prototype's four-layer sequence into `globals.css`. CRT field now mounts during both `phase === 'boot'` and `phase === 'on'` so animations have a render target.

**Tests:** 1 assertion flip — CRT field now expected in DOM during boot.

### Pass 2: Close divergences from the prototype (`1ca8233`)

**What:** Evaluator + product reviewer on pass 1 flagged three issues: (a) the paint-down curtain snapped at phase transition without a graceful fade; (b) the CRT field had no z-index, so `position: relative` tabs and body painted on top of it — scanlines/flicker didn't cover those regions; (c) no tests verified the new four-layer structure.

**Fix (user chose "option C" from my proposal):** Split `.crt-ambient` out as a sibling of the CRT field with no z-index. It paints in stacking-context step 6 (positioned z:auto) — ahead of the in-flow header but behind the positioned tabs/body in DOM order — which deliberately scopes the amber blend to the header surface (preserving the user's liked effect). The boot sequence + scanlines got `z-index: 3` in a separate wrapper so they cover ALL overlay content. Ported the prototype's `[data-phase='on'] .crt-bloom { opacity: 0; clip-path: inset(0 0 0% 0); transition: opacity 200ms ease 80ms }` rule so the curtain dissolves rather than snaps. Introduced `--u-paper: #0d0b0a` CSS variable in `:root` (mirrored from `tailwind.config.ts` `u-paper.DEFAULT`) to replace hardcoded hex. Reordered DOM children to match prototype: flicker → bloom → scanlines.

**Tests:** +4 assertions covering ambient-as-sibling structure, all three boot layers inside the z:3 wrapper, ambient mount during boot, tab-flash wrapper presence on body pane.

### Pass 3: Once-per-session boot (`96a2f73`) — Quality Improvement

**What:** User requested that the boot fire only on the first overlay open of a browser session, not every open. Rationale: a boot gesture is an editorial reveal, not a repeated effect. Classified as Quality Improvement per the `/change` skill — spec update comes first. User chose `sessionStorage` scope (per-tab) over `localStorage` (persistent) or page-lifetime ref.

**Fix:** Updated `docs/REQUIREMENTS.md` D9 and `docs/PHASE_STATUS.md` deliverable-9 bullet first to capture the new semantics. Added `BOOT_SESSION_KEY = 'iampatterson.overlay.booted'` with `hasBootedThisSession` / `markBootedThisSession` helpers wrapping the `sessionStorage` access. The open-effect now checks the session flag alongside `prefers-reduced-motion` before deciding between boot and on. Try/catch around storage access so strict-privacy browsers fall back to the old boot-every-open behavior rather than breaking the overlay. Flag is set when boot *starts* (not completes) so an interrupted boot still counts as booted.

**Tests:** +2 new cases (skip-on-prior-boot, flag-gets-set). 1 existing integration test reshaped from "close-then-reopen re-boots" to "close-then-reopen skips boot per once-per-session spec." `sessionStorage.clear()` added to `beforeEach` in both overlay test files; defensive `jest.useRealTimers()` added to the integration test's `afterEach` to prevent fake-timer leakage between tests.

## Issues & Technical Debt

Both reviewers flagged a small set of unresolved issues. User did not explicitly re-accept UAT after this session's work; treat these as follow-up items if 9A-redesign merges without further iteration.

1. **Header and tabs remain in the DOM during `phase: 'boot'`** — Minor (product reviewer). The prototype's `.overlay-panel > * { opacity: 0 }` rule would have hidden them in opacity too; the implementation uses `[data-phase='boot'] .tab-flash { opacity: 0 }` which only hides the tab-flash wrapper inside the body. The z:3 bloom curtain visually masks the header/tabs during boot, but on a low-end device with paint-jank the curtain could miss a frame and the header/tabs would flash briefly. Resolution: either add a `[data-phase='boot']` rule to hide header/tabs explicitly, or add a code comment acknowledging the deviation.

2. **No replay affordance for the boot gesture** — Minor (product reviewer). Once a visitor has seen boot, they'd need to open a new tab to see it again. For a portfolio site whose thesis is "see how it works," a dev-mode reset or debug hook might matter. Low priority; flagging for consideration.

3. **Boot-once flag not set under `prefers-reduced-motion`** — Minor (evaluator). A user with reduced-motion enabled who later disables it and re-opens in the same session would get the full boot sequence (flag was never set on the reduced-motion path). Reduced-motion is typically persistent across a session, so this is edge-case-only. Location: `src/components/overlay/under-the-hood-view.tsx` — the reduced-motion branch of the open-effect returns before `markBootedThisSession()`.

4. **`.crt-bloom` is missing `transform-origin: top`** — Minor (evaluator). Present in prototype at `overlay.css:76`, not ported. No visible effect today since `clip-path` carries the motion; fidelity-only divergence.

5. **Thin animation-behavior assertions** — Minor (evaluator). Tests verify DOM structure and phase-state transitions but don't assert the actual CSS effects apply (e.g., `getComputedStyle(...).opacity` checks on the tab-flash wrapper during boot). Partially addressed this session but could go deeper.

6. **TDD discipline lapsed on commits 1 and 2** — Minor (evaluator). Tests in those commits were added in the same commit as the implementation change rather than red-first. Commit 3 *was* written red-first (red state explicitly confirmed before implementation). CLAUDE.md's "non-negotiable" TDD requirement was partially honored.

## Session Notes

- **Stacking-context trick is load-bearing.** The split-layer design (`crt-ambient` at z:auto, `crt-field` at z:3) exploits the CSS stacking-context rule that positioned z:auto elements paint after in-flow block children but before later-DOM-order positioned siblings. This is what scopes the amber blend to the header. The test suite has an assertion that `crtField.contains(ambient) === false` — that guard is there deliberately to prevent a future refactor from flattening the structure back under one wrapper. If that test ever needs to change, also re-read `src/components/overlay/under-the-hood-view.tsx:179-184` for the original rationale.
- **sessionStorage key namespacing.** Project-wide key prefix is `iampatterson.` (dot-namespaced). Match this pattern for any future browser-storage keys.
- **CSS variable discipline.** `--u-paper` in `:root` now mirrors the Tailwind `u-paper.DEFAULT` token — declared in `src/styles/globals.css` with a comment pointing to `tailwind.config.ts:107` as the source of truth. If `u-paper.DEFAULT` ever changes, update both. Consider a CSS-to-Tailwind sync pass if this pattern expands to other underside tokens.
- **Four-commit session pattern.** The work progressed via observation → fix → review → refine → spec change → refine → cleanup. Not a single ship, but every commit left the codebase in a shippable state (tests green, build clean). Worth preserving this rhythm: small targeted commits with reviewer feedback between each step beats a monolithic "all the fixes" PR.
