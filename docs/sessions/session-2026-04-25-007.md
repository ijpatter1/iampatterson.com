### Phase 1 D3 architectural amendment — Cookiebot manual blockingmode + explicit gtag bridge (`518e1e1`)

Discovered mid-session via D9 implementation. The dev server was throwing a React-19/Next-16 hydration mismatch on every page load — initially incorrectly attributed to D9 by me, then empirically ruled out via a TEMP D9 revert (`4f7eeb5`) that confirmed the warning persisted without D9. Three failed-fix attempts before the real fix landed:

1. **Failed `494bd75` → reverted `2d3fa95`:** refactored CookiebotScript + the GTM consent-defaults block as raw `<script>` tags to drop `next/script + beforeInteractive`. Made the diagnostic clearer (revealed Cookiebot's `data-blockingmode="auto"` actively rewrites head DOM during script execution) but did not fix the warning, and introduced a new "Scripts inside React components are never executed when rendering on the client" warning.
2. **Failed `b2147da` → reverted `aae67a8`:** added `suppressHydrationWarning` to the `<head>` element. Investigation revealed the attribute is one DOM-element deep — it suppresses warnings on the element it's set on but not on nested element trees, so the script-tag mismatches three component-levels below were unaffected.
3. **Real fix `518e1e1`:** root-cause architectural change. Cookiebot loaded with `data-blockingmode="manual"` instead of `"auto"`, which stops Cookiebot from rewriting the head DOM. Added new `bridgeToGtagConsent` helper in `src/lib/events/track.ts` that calls `window.gtag('consent', 'update', {...})` with the explicit Cookiebot→gtag-signal mapping (statistics→analytics_storage; marketing→ad_storage+ad_user_data+ad_personalization; preferences→functionality_storage+personalization_storage). What auto-mode used to do implicitly is now explicit. Wired into both `trackConsentUpdate` (new accept/decline) and `initConsentState` (returning visitors). Verified the GTM container (`infrastructure/gtm/web-container.json`) is fully Consent Mode v2 gated (every analytics tag has `consentSettings: { analytics_storage: required }`), so the gating remains correct — just delegated entirely to GTM Consent Mode rather than belt-and-suspenders Cookiebot script-rewriting.

User confirmed the fix on dev refresh: warning gone. Cookiebot's "WARNING! Internal test version. Cookie Consent not logged." messages still print on localhost — those are Cookiebot's own dev-mode notices unrelated to hydration; expected and harmless.

Doc updates landed alongside per the change-skill quality-improvement workflow: ARCHITECTURE.md "Cookiebot + GTM Consent Mode" section rewritten with the new integration sequence + signal mapping; REQUIREMENTS.md Phase 1 D3 amendment note; PHASE_STATUS.md 10d ledger entry recording the amendment.

### Architecture decisions

- **Cookiebot blockingmode = "manual"** with explicit `bridgeToGtagConsent`. The blockingmode="auto" pattern is from before Consent Mode v2 was widely supported — a belt-and-suspenders safety net for sites that don't fully implement Consent Mode. We do, so manual is the cleaner architecture. The bridge function is the explicit replacement for what auto-mode did implicitly. ARCHITECTURE.md "Cookiebot + GTM Consent Mode" section rewritten to describe the new sequence end-to-end.
- **D9 sequencing inverted.** Originally D7 → D9; user direction during session-007 swapped it: D9 builds the inspector surface, then D7 reduces to a one-line classifier entry consuming it. Sequencing decision saved D7 from needing its own dedicated Overview-portal surface code.
- **`useStorageInspector` lives in `OverlayView`, snapshot threaded down to `OverviewTab` + `ConsentView`.** Single source of truth; both tabs read off the same snapshot reference. `ConsentView` and `OverviewTab` accept optional `storage?: StorageSnapshot` props with empty-snapshot defaults, keeping them presentational and testable without `OverlayProvider` wrappers.
- **`getAnonymousId()` returns `''` on SSR rather than throwing.** Callers treat empty as "not yet known" — same convention as `getSessionId()` already established in Phase 2. The warehouse-write substitution checks `aid.length > 0` before substituting, falling back to `NULL` seed otherwise. This is the SSR-safety contract; deviating would surface as a hydration mismatch.
- **`initConsentState` is silent (no data-layer event).** Returning visitors don't get a phantom `consent_update` in their Timeline for a consent decision they didn't actively make this session. The bridge call to `gtag('consent', 'update', ...)` still happens (so Consent Mode v2 sees the right state); only the Timeline-visible event is suppressed.
