## Session Notes

### Patterns established / reinforced this session

- **Visual image identification for schema-driven catalogues.** D8.f's image-to-product mapping was genuinely opaque at the filename level (DSC-prefixed + UUID-prefixed filenames carried no semantic hint). Reading each webp via the Read tool gave a visual identification that made the mapping obvious for 5/6 pairs and best-guess for the sixth. Pattern worth repeating: when you have a schema of N items and N unnamed assets, visually inspect rather than infer from filenames — spend 2 minutes on the mapping to avoid mis-pairing that ships silently.
- **Reconciling doc drift in place.** D8.f surfaced two doc-level inaccuracies (wrong path + wrong count). The prior handoff flagged them forward with a "to be reconciled when D8.f ships" note rather than editing REQUIREMENTS mid-flight. This session reconciled them in the D8.f commit + docs sweep, not in a separate "doc fix" commit. Pattern: when a deliverable surfaces a doc-drift issue, fix the docs in the same commit as the deliverable — keeps the commit message explaining *why* the docs changed.
- **Cluster commits vs per-sub-item commits.** Homepage cluster shipped as 4 individual commits (one per sub-item); overlay cluster shipped as a single commit (h+i+j together). Driver: overlay sub-items shared the palette-token addition + touched related files, so splitting them would have created merge-conflict risk on the shared `tailwind.config.ts` + `consent-view.tsx` changes. Pattern: split when sub-items are genuinely independent; bundle when they share substrate.
- **Test-pin updates on verb renames.** D8.j renamed "Active destinations" → "Firing destinations" and "Suppressed destinations" → "Blocked destinations" in the Consent tab. Both existing tests' regex assertions had to be walked forward. Pattern: when a rename is semantic (matching schema terms like `sent` and `blocked_consent`), update the test copy alongside the source; the comment-in-test explaining *why* the rename happened makes the test self-documenting for future readers.

### Architecture decisions

- **D8.f: palette triplet stays in the schema.** Removing `palette` would have required removing the cart-line thumbnail's fallback background (the thumbnail renders a 48×48 box with `style={{ background: p.palette[0] }}` behind the `<Image>`; if the image CDN is slow or fails, the box stays coloured rather than showing a white rectangle). Keeping the triplet costs 3 hex values per SKU in the seed array; removing it would have broken the skeleton-behind-image pattern. Also leaves room for palette-driven theming on future product-card hover states without a schema change.
- **D8.j: glyphs as Unicode, not inline SVG.** The ✓/× Unicode characters render as ~9-byte text additions to the component source. SVG icons would have added either a new icon component or inline `<svg>` markup at 5-10 lines per call site. For a polish item on an overlay that already reads as a terminal-ish surface (Cascadia/JetBrains Mono throughout), Unicode glyphs fit the aesthetic and carry a small cross-platform rendering risk (flagged above under "honest gaps"). If the risk materializes, swap to SVG icons in a follow-up.
- **D8.h: directive block as its own `<section>` with `data-testid`.** The testid lets the F2 structural test specifically address the new block without a brittle "first child at position 0" pin. Pattern worth continuing for other overlay tabs if more sections land — explicit testids on tab-level sections make document-order invariants expressible without relying on DOM index.
- **D8.i: directive sentence appended rather than a separate pointer chip.** Considered a dedicated "Manage consent" chip / button that calls `window.Cookiebot.renew()` directly. Rejected for this session because: (a) the chip would blur the "what this tab shows" directive with "what action you can take" control, (b) `window.Cookiebot.renew()` requires Cookiebot script readiness gating, (c) the existing Cookiebot badge in the bottom-left *is* the in-house control — pointing at it preserves that surface as the canonical action point. If post-evaluator feedback says a chip is warranted, the renew-button pattern is a straightforward follow-up.
