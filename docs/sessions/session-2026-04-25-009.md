### Architecture decisions

- **D2 fallback-shape parity.** Both `DashboardPayoff` fallback variants (null-signer + load-timeout) use identical InlineDiagnostic chrome + identical second-sentence shape ("The dashboard lives at [link] behind Google SSO."). Two error states, one resolution shape. Visitor experiences both as one degradation mode, not two. Was the right design call per Pass-2 product reviewer's 5.00/5 framing.
- **D3 page_engagement as a 1Hz tick + Page Visibility gate.** Considered alternatives: setTimeout chain with millisecond timestamps (more accurate but more complex), monkey-patched intersection observer for "engaged" definition (overkill). Settled on the pattern that mirrors `ScrollDepthTracker`: simple useEffect with setInterval, visibility-event-listener for pause/resume, pathname dep for SPA-route reset. The 1Hz tick is cheap when visible, zero-cost when hidden.
- **D6 in-process load tests over real Cloud Run preview.** The hot paths in both services are CPU-bound (`parsePubSubMessage` is base64 decode + JSON parse + field validation; `generateDateRange` is event synthesis). Stressing them in-process at high N produces the throughput ceiling without network-layer noise. Real-network load testing is captured as a Phase 11 D3 carry-forward — different concern (HTTPS termination, OIDC verification, autoscaling latency) at a different layer.
- **`page_engagement` BQ columns land now, sGTM trigger deferred.** Mirrors the WebVital precedent's "schema lands with the event, trigger lands later" sequencing. Avoids the silent-payload-drop failure mode that would occur if the trigger were wired against an `events_raw` table without the columns. Pass-2 fix-pack closure.
