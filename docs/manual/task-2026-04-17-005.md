## Steps

### 1. Re-run `setup-domain.sh` to apply the URL-map split

Session 021 added step 8 to `infrastructure/metabase/setup-domain.sh` — creates `metabase-backend-direct` (non-IAP backend, same serverless NEG) and a path matcher on the existing URL map routing `/api/*` and `/embed/*` to it. Idempotent — all prior LB components are skipped.

```bash
cd /workspace/infrastructure/metabase
./setup-domain.sh --no-wait 2>&1 | tee /tmp/metabase-task5a.log
```

Expected output near the end:

```
==> 8/8 Non-IAP backend + URL-map split...
+ gcloud compute backend-services create metabase-backend-direct ...
+ gcloud compute backend-services add-backend metabase-backend-direct ...
+ gcloud compute url-maps add-path-matcher metabase-url-map ...
```

Verify from the shell you ran it in:

```bash
# UI path still IAP-gated
curl -sI https://bi.iampatterson.com/ | head -3
# expect: HTTP/2 302  Location: https://accounts.google.com/...

# API path now bypasses IAP
curl -sI https://bi.iampatterson.com/api/health | head -3
# expect: HTTP/2 200  (direct from Metabase)

# Embed path also bypasses IAP (returns 4xx, not 302)
curl -sI https://bi.iampatterson.com/embed/question/not-a-jwt | head -3
# expect: HTTP/2 4xx  (Metabase rejects the bad JWT; NOT a 302 to Google)
```

If the 4xx on `/embed/*` comes back as `302 Location: accounts.google.com`, the path matcher didn't land — re-read the `8/8` output in the log for an error, or inspect via:

```bash
gcloud compute url-maps describe metabase-url-map --global --project=iampatterson \
  --format='value(pathMatchers.name)'
# expect: direct-paths
```

### 2. Generate an admin API key in Metabase

1. Open `https://bi.iampatterson.com/admin/settings/authentication/api-keys` (log in via IAP if prompted).
2. Click **Create API Key**.
3. **Key Name:** `apply-sh` (or similar — purely descriptive).
4. **Group:** `Admin`.
5. Click **Create**. Copy the key value — it is shown ONCE.

### 3. Store the key in Secret Manager

```bash
# Replace <KEY> with the value you just copied
echo -n "<KEY>" | gcloud secrets create metabase-api-key \
  --project=iampatterson \
  --replication-policy="automatic" \
  --data-file=-
```

Grant your own account access to read it (so `apply.sh` can fetch it):

```bash
gcloud secrets add-iam-policy-binding metabase-api-key \
  --project=iampatterson \
  --member="user:$(gcloud config get-value account)" \
  --role="roles/secretmanager.secretAccessor"
```

## Rollback

If something goes wrong:

- **Delete a stray card:** Metabase UI → card → three-dot menu → Archive.
- **Delete the entire collection:** Metabase UI → E-Commerce Dashboards collection → three-dot menu → Archive. On next `apply.sh` run, the script recreates everything from the specs.
- **Revoke the API key:** Metabase UI → Admin → Authentication → API Keys → delete. Then delete the Secret Manager secret.
- **Remove the URL-map split:** `gcloud compute url-maps remove-path-matcher metabase-url-map --path-matcher-name=direct-paths --global --project=iampatterson`, then `gcloud compute backend-services delete metabase-backend-direct --global --project=iampatterson`. `setup-domain.sh` re-adds both on next run.
